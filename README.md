# threejs-pyside6

A PySide6 desktop app hosting Three.js scenes via `QWebEngineView`. Reference renderer for Silkroad Online zones, with a navmesh edge viewer and pathfinding overlays + entity layers on the way.

## Status

- ✅ **Phase 0** — render-stack scaffolding (procedural terrain, throughput bench, Python↔JS bridge).
- ✅ **Phase 1** — Hotan Kingdom zone visual: 7 regions, ~458 building submeshes, ~196 decorations, ~330 textures.
- ✅ **Phase 1.5** — standalone SRO navmesh edge viewer (`navmesh.py`): region 5c87, 57 edge groups, interactive PySide6 left panel. See [`docs/NAVMESH.md`](docs/NAVMESH.md).
- 🔜 **Phase 2** — navmesh overlays *inside the zone view*. See [`docs/NAVMESH_PLAN.md`](docs/NAVMESH_PLAN.md).

## Quickstart

```bash
uv sync

# Procedural scaffolding (no SRO data required)
uv run python main.py        # procedural terrain demo
uv run python bench.py       # render-throughput benchmark
uv run python bridge.py      # Python <-> JS bridge with InstancedMesh agents
uv run python sweep.py       # parameter sweep over bridge.py configs

# Hotan Kingdom SRO zone (one-shot preprocess, then run)
uv run python scripts/build_zone.py
uv run python zone.py

# SRO navmesh edge viewer (needs an extracted PK2 tree at /Users/hodung/Workspace/silkroad/sro-data
# or override with --map-root / --data-root / NAVMESH_MAP_ROOT / NAVMESH_DATA_ROOT)
uv run python navmesh.py
```

`bridge.py` takes `--n <entities>`, `--hz <target rate>`, `--protocol {json,bytes}`, `--seconds <auto-quit>`.

`scripts/build_zone.py` takes `--zone`, `--regions`, `--center`, `--silknav-out`, `--web-root`, `--no-o2`. Defaults render the 7-region Hotan Kingdom zone from the vendored `data/silknav_export/`.

`navmesh.py` takes `--nvm`, `--map-root`, `--data-root`. Defaults load region 5c87.

## Layout

- `main.py`, `bench.py`, `bridge.py`, `sweep.py` — Phase 0 scaffolding.
- `zone.py` — Hotan Kingdom zone launcher (Phase 1).
- `navmesh.py` — SRO navmesh edge viewer (Phase 1.5).
- `parsers/` — direct ports from [silk-nav](https://github.com/dudaka/silk-nav): `.nvm` / `.bsr` / `.cpd` / `.bms` / `Object.ifo`. Used by `navmesh.py`.
- `scripts/build_zone.py` — preprocessor: DDS→PNG via Pillow, manifest emission.
- `data/silknav_export/` — vendored copy of silk-nav's Blender-pipeline output (~85 MB, tracked).
- `web/index.html`, `web/bench.html`, `web/bridge.html` — Phase 0 scenes.
- `web/zone.html` — Hotan Kingdom Three.js scene.
- `web/navmesh.html` — navmesh viewer Three.js scene.
- `web/zones/<zone>/` — generated by `build_zone.py` (~102 MB, gitignored, regenerable).
- `web/vendor/three/` — Three.js r184.
- `web/vendor/qwebchannel/` — `qwebchannel.js` (used by `bridge.py` only).

## Dependencies

- Python ≥ 3.11 (3.13 resolves in `uv.lock`).
- PySide6 ≥ 6.8 (brings QtWebEngine).
- NumPy ≥ 2.0.
- Pillow ≥ 12 (DDS → PNG; for the zone preprocessor).

The SRO data side is owned by [silk-nav](https://github.com/dudaka/silk-nav). For the zone demo, we vendor its export output rather than depending on its working tree. For the navmesh viewer, we ported its parsers directly into `parsers/` and read the live PK2 dump tree.
